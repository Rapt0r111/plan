## Вердикт: FastAPI не нужен. Вот почему.

Ваш текущий стек — один из лучших для 2026–2027 года. Bun + Next.js 16 + Drizzle + SQLite — это не легаси, это передовая архитектура. Смена стека принесёт боль без выгоды. Сначала объясню логику, потом покажу что реально стоит сделать.

### Почему SQLite лучше PostgreSQL для вашего случая

SQLite с WAL-режимом (у вас уже включён) показывает задержку менее 0.5 мс на локальной FS. PostgreSQL добавит сетевой round-trip к каждому запросу даже на localhost. При 7 пользователях это регрессия, а не улучшение. SQLite переживает "ренессанс" в 2025–2026: Turso (LibSQL) дал ему edge-синхронизацию, Cloudflare D1 строит на нём облако. Это не умирающий инструмент — это embedded-база, которую наконец оценили по достоинству.

PostgreSQL имеет смысл при 50+ одновременных пользователях, JSONB-колонках, сложном full-text search или репликации. У вас этого нет и не будет.

### Почему не FastAPI

FastAPI — отличный фреймворк, но он решает не ваши проблемы. Добавление Python-бэкенда означает: два рантайма (Node/Bun + Python), HTTP-прослойку между Next.js и API, дублирование типов (Pydantic на сервере, TypeScript на клиенте), отдельный деплой, отдельный мониторинг. Это архитектура для команды из 5+ человек с Python-разработчиками.

FastAPI оправдан в трёх случаях: нужен ML-инференс (Python-экосистема незаменима), нужны сложные фоновые задачи с Celery, или у вас > 200 RPS с нетривиальной бизнес-логикой. У вас 7 военнослужащих, не 200 RPS.

### Что реально стоит исправить прямо сейчас

В текущем коде есть несколько серьёзных ошибок, которые дадут больше выгоды при исправлении, чем любая миграция стека.

**Баг #1 — `revalidateTag` неправильно вызван повсюду.** В 15+ местах кода написано `revalidateTag(EPICS_CACHE_TAG, "max")`. Второй аргумент не существует в Next.js API — функция принимает только один параметр. Это тихий баг: код не падает, но и не работает как ожидается.

```typescript
// БЫЛО (неверно — 15+ мест в коде):
revalidateTag(EPICS_CACHE_TAG, "max");

// ДОЛЖНО БЫТЬ:
revalidateTag(EPICS_CACHE_TAG);
```

**Баг #2 — Нет аутентификации.** Все API-маршруты (`/api/tasks`, `/api/epics`, `/api/users`) открыты без авторизации. Любой в сети может создавать/удалять данные. Для интранета минимум — проверка сессии или JWT.

**Баг #3 — Нет rate limiting.** `POST /api/tasks` можно вызывать бесконечно, заполнив БД мусором.

**Анти-паттерн #4 — `getAllEpicsWithTasks` дважды на дашборде.** В `dashboard/page.tsx` вы делаете `getAllEpics()` + `getAllEpicsWithTasks()` параллельно. Первый запрос включён во второй — это лишний SQL без выгоды.

### Эволюционный путь: что добавить в 2026–2027

**Шаг 1 (сейчас): починить баги выше.** Возврат: безопасность + корректная работа кэша.

**Шаг 2 (если нужен real-time): Bun WebSockets.** Bun имеет нативный WebSocket-сервер без дополнительных зависимостей. Можно добавить live-обновления доски без polling.

```typescript
// shared/server/ws.ts — Bun native WebSocket
const server = Bun.serve({
  websocket: {
    message(ws, data) { ws.publish("tasks", data); },
  },
});
```

**Шаг 3 (если вырастете до 20+ пользователей): LibSQL вместо SQLite.** LibSQL — форк SQLite с embedded-репликацией. Миграция Drizzle минимальная: поменяется только `client.ts`, схема и запросы остаются идентичными.

```typescript
// Было:
import { Database } from "bun:sqlite";
const sqlite = new Database(DB_PATH);

// Станет:
import { createClient } from "@libsql/client";
const client = createClient({ url: "file:local.db" }); // или turso://...
```

**Шаг 4 (если нужен тип-безопасный API для будущих клиентов): tRPC v11.** Если появится мобильное приложение или внешний клиент, tRPC даст end-to-end типизацию без генерации кода. Это не замена Route Handlers — это слой поверх них.


### Производительность текущего стека в цифрах (2026)

Bun SQLite через `bun:sqlite` выдаёт порядка 200 000 простых запросов в секунду на современном железе. Ваш дашборд делает 2–3 SQL-запроса. При 7 пользователях это 0.0001% от пропускной способности. PostgreSQL + network overhead на localhost — около 10 000 запросов в секунду. Вы уже в 20 раз быстрее, чем могли бы быть.

### Итог

Текущий стек правильный для 2026–2027. Миграция на FastAPI + PostgreSQL для 7 пользователей — это решение несуществующей проблемы с реальными затратами. Вместо этого: исправьте `revalidateTag`, добавьте аутентификацию, и рассмотрите LibSQL когда/если масштаб вырастет. Сэкономите несколько недель работы и получите более стабильную систему.
