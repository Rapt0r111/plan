
  # План дальнейшего развития TaskFlow

  ## 1. Цель развития

  TaskFlow уже закрывает базовые сценарии управления задачами: доска, эпики, оперативные задачи, недельный план, “Мой день”, роли, аудит, настройки, PWA/
  offline-first и live-обновления.

  Дальнейшее развитие должно быть направлено не на добавление сложных декоративных функций, а на повышение удобства ежедневной работы:

  - пользователь должен быстрее понимать, что ему делать сейчас;
  - руководитель должен видеть риски без ручного сбора информации;
  - администратор должен управлять системой без вмешательства в код или БД;
  - интерфейс должен оставаться быстрым на старых ноутбуках без GPU;
  - система должна оставаться интранет/offline-first без внешних сервисов.

  ———

  ## 2. Этап 1 — быстрые UX-улучшения

  ### 2.1. Сделать /today главным рабочим экраном

  Необходимо превратить экран “Сегодня / Мой день” в основной центр работы пользователя.

  Реализовать:

  - единый список “Что делать сейчас”;
  - объединение задач из:
      - назначенных задач;
      - оперативных задач;
      - недельного плана;
      - просроченных задач;
      - задач с приближающимся сроком;
      - важных задач без движения;

  - приоритетную сортировку:
      1. просрочено;
      2. срок сегодня;
      3. высокий приоритет;
      4. задача заблокирована;
      5. давно не обновлялась;

  - быстрые действия прямо из списка:
      - открыть задачу;
      - завершить;
      - перенести срок;
      - изменить статус;
      - добавить комментарий;
      - отложить на завтра;

  - визуально разделить:
      - “Срочно”;
      - “Сегодня”;
      - “Можно позже”;
      - “Ожидает других”.

  Цель: пользователь заходит утром и сразу видит рабочий план без перехода по разным разделам.

  **Статус на 2026-06-01: реализовано.**

  Реализовано:
  - `/today` показывает единый список “Что делать сейчас” с задачами доски, оперативными задачами и пунктами недельного плана;
  - личный список включает назначенные задачи, просрочки, задачи на сегодня, приближающиеся сроки, высокий приоритет, блокировки и задачи без движения;
  - сортировка закреплена в коде: просрочено → срок сегодня → высокий приоритет → заблокировано → давно не обновлялась → ближайший срок → назначено;
  - список визуально разделен на “Срочно”, “Сегодня”, “Можно позже”, “Ожидает других”;
  - из списка доступны быстрые действия: открыть, завершить, перенести срок, сменить статус, добавить комментарий, отложить на завтра;
  - для недельного плана из списка доступны открыть и завершить текущий пункт; изменение расписания остается через экран недельного плана.

  Осталось:
  - если потребуется полное действие “перенести/комментировать отдельное повторение недельного плана”, нужна отдельная модель разовых исключений/комментариев к occurrence, а не изменение всего шаблона.

  Что впереди:
  - следующий пункт плана — 2.2 “Сохраненные представления” для `/today`, доски, оперативки, управления и недельного плана.

  ———

  ### 2.2. Сохраненные представления

  Добавить возможность сохранять пользовательские фильтры и виды.

  Где нужно:

  - доска задач;
  - оперативные задачи;
  - управление;
  - “Мой день”;
  - недельный план.

  Реализовать:

  - сохранение фильтров;
  - сохранение сортировки;
  - сохранение группировки;
  - выбор вида по умолчанию;
  - быстрый переключатель сохраненных видов;
  - личные виды пользователя;
  - системные виды для всех.

  Примеры видов:

  - “Мои задачи”;
  - “Просроченные”;
  - “Сегодня”;
  - “Без ответственного”;
  - “Оперативка моей группы”;
  - “Риски недели”;
  - “Задачи без движения”.

  ———

  ### 2.3. Улучшить пустые состояния

  Во всех экранах, где может быть пустой список, добавить полезные подсказки.

  Реализовать:

  - объяснение, почему список пуст;
  - кнопку основного действия;
  - короткую подсказку, что делать дальше;
  - ссылку на справку или Command Palette.

  Примеры:

  - если нет задач — “Создайте первую задачу”;
  - если нет оперативных задач — “Добавьте оперативную задачу для группы”;
  - если нет эпиков — “Создайте эпик для группировки задач”;
  - если фильтр ничего не нашел — “Сбросить фильтры”.

  ———

  ### 2.4. Встроенная справка и горячие клавиши

  Добавить простой раздел помощи внутри интерфейса.

  Реализовать:

  - пункт “Справка” в Command Palette;
  - клавишу ? для открытия подсказок;
  - список горячих клавиш;
  - описание основных сценариев:
      - как создать задачу;
      - как назначить исполнителя;
      - как работать с оперативкой;
      - как пользоваться “Моим днем”;
      - как искать задачи;
      - как менять тему и производительность;

  - короткие подсказки на сложных экранах.

  ———

  ### 2.5. Performance-first UI

  Закрепить правило: интерфейс должен быть быстрым даже на слабом устройстве.

  Реализовать:

  - минимальные анимации по умолчанию;
  - отключение декоративных эффектов в performance mode;
  - отсутствие обязательных blur/glass/ambient/cursor-эффектов;
  - поддержку prefers-reduced-motion;
  - понятные focus/hover состояния без тяжелой графики;
  - настройку “Экономичный режим интерфейса”;
  - проверку основных экранов на слабых устройствах.

  Не добавлять новые тяжелые анимации без функциональной пользы.

  ———

  ## 3. Этап 2 — снижение ручной рутины

  ### 3.1. Actionable-уведомления

  Уведомления должны быть не просто списком событий, а рабочим инструментом.

  Реализовать:

  - центр уведомлений;
  - разделение уведомлений:
      - новые задачи;
      - изменения сроков;
      - комментарии;
      - упоминания;
      - просрочки;
      - системные события;

  - быстрые действия:
      - открыть;
      - отметить прочитанным;
      - отложить;
      - завершить;
      - сменить срок;
      - перейти к обсуждению;

  - счетчик непрочитанных уведомлений;
  - настройку типов уведомлений.

  ———

  ### 3.2. “Что изменилось с последнего входа”

  Добавить экран или блок, который показывает важные изменения с момента последнего посещения.

  Показывать:

  - новые назначенные задачи;
  - измененные сроки;
  - новые комментарии;
  - смену статуса;
  - задачи, ставшие просроченными;
  - новые оперативные поручения;
  - изменения в эпиках.

  Цель: пользователь не должен вручную искать, что произошло.

  ———

  ### 3.3. Шаблоны задач

  Добавить типовые шаблоны для повторяющихся задач.

  Реализовать:

  - создание шаблона из задачи;
  - выбор шаблона при создании задачи;
  - поля шаблона:
      - название;
      - описание;
      - чеклист;
      - приоритет;
      - типовой срок;
      - исполнитель или группа;
      - эпик;

  - шаблоны личные и общие;
  - быстрый запуск шаблона из Command Palette.

  Примеры:

  - “Еженедельный отчет”;
  - “Проверка оборудования”;
  - “Подготовка документа”;
  - “Оперативное поручение”.

  ———

  ### 3.4. Чеклисты внутри задач

  Добавить простые чеклисты для задач, где есть несколько подэтапов.

  Реализовать:

  - список пунктов;
  - отметку выполнено/не выполнено;
  - прогресс выполнения;
  - изменение порядка пунктов;
  - добавление пункта из шаблона;
  - отображение прогресса на карточке задачи.

  Не заменять чеклистами подзадачи полностью — чеклист нужен для легких внутренних шагов.

  ———

  ### 3.5. Массовые действия

  Добавить массовое редактирование задач.

  Реализовать:

  - выбор нескольких задач;
  - массовое изменение:
      - статуса;
      - срока;
      - исполнителя;
      - приоритета;
      - эпика;
      - группы;

  - подтверждение перед применением;
  - отображение количества затронутых задач;
  - запись в аудит.

  Особенно важно для руководителей и администраторов.

  ———

  ### 3.6. Улучшенный глобальный поиск

  Расширить поиск и Command Palette.

  Искать по:

  - задачам;
  - эпикам;
  - пользователям;
  - комментариям;
  - оперативным задачам;
  - недельному плану;
  - настройкам;
  - сохраненным видам.

  Добавить фильтры:

  - статус;
  - исполнитель;
  - срок;
  - эпик;
  - приоритет;
  - группа;
  - просрочено/не просрочено.

  ## 4. Этап 3 — отчеты, импорт и эксплуатация

  ### 4.1. Отчеты без внешних сервисов

  Добавить встроенные отчеты.

  Реализовать отчеты:

  - недельная сводка;
  - выполненные задачи;
  - просроченные задачи;
  - нагрузка по пользователям;
  - нагрузка по группам;
  - задачи без движения;
  - задачи без ответственного;
  - причины блокировок;
  - выполнение по эпикам.

  Форматы:

  - просмотр внутри системы;
  - экспорт CSV;
  - экспорт PDF или печатная версия;
  - копирование краткой сводки в буфер.

  ———

  ### 4.2. Причины блокировки и риска

  Добавить к задачам структурированное поле причины проблемы.

  Реализовать:

  - признак “заблокировано”;
  - причина блокировки:
      - нет данных;
      - ожидает другого сотрудника;
      - техническая проблема;
      - нет доступа;
      - внешний фактор;
      - другое;

  - комментарий к блокировке;
  - дата установки блокировки;
  - отображение блокировки на карточке;
  - отчет по причинам блокировок.

  Цель: система должна показывать не только факт просрочки, но и причину.

  ———

  ### 4.3. Импорт из файлов

  Добавить импорт данных без внешних интеграций.

  Поддержать:

  - пользователей;
  - групп;
  - задач;
  - эпиков;
  - типовых шаблонов.

  Формат:

  - CSV как основной;
  - XLSX можно оставить как будущую опцию.

  Обязательно реализовать dry-run:

  1. пользователь загружает файл;
  2. система показывает найденные строки;
  3. система показывает ошибки;
  4. пользователь подтверждает импорт;
  5. только после подтверждения данные записываются.

  ———

  ### 4.4. Backup / Restore UI

  Добавить админский экран для обслуживания системы.

  Реализовать:

  - создание резервной копии;
  - скачивание резервной копии;
  - просмотр даты последнего backup;
  - размер базы;
  - предупреждение перед restore;
  - restore только для администратора;
  - запись всех действий в аудит.

  ———

  ### 4.5. Улучшенный аудит

  Сделать аудит удобным рабочим инструментом, а не просто техническим журналом.

  Реализовать:

  - фильтр по пользователю;
  - фильтр по сущности;
  - фильтр по типу действия;
  - фильтр по периоду;
  - поиск по тексту;
  - экспорт аудита;
  - человекочитаемые описания действий;
  - группировку по дате.

  ———

  ## 7. Тестирование

  ### Unit tests

  Добавить тесты для:

  - сортировки задач в /today;
  - сохраненных представлений;
  - фильтров;
  - уведомлений;
  - шаблонов задач;
  - чеклистов;
  - dry-run импорта;
  - прав доступа.

  ### E2E tests

  Проверить сценарии:

  - пользователь открывает /today и понимает план работы;
  - пользователь сохраняет вид доски;
  - пользователь создает задачу из шаблона;
  - пользователь выполняет массовое действие;
  - пользователь открывает уведомление и выполняет quick action;
  - администратор экспортирует отчет;
  - администратор делает backup;
  - пользователь с ограниченными правами не видит чужие данные.

  ### Performance tests

  Проверить:

  - доска не лагает при большом количестве задач;
  - отключенные анимации действительно не выполняются;
  - интерфейс работает с prefers-reduced-motion;
  - тяжелые визуальные эффекты не включены по умолчанию.

  ———

  ## 8. Правила дальнейшей разработки

  - Не добавлять внешние сервисы без отдельного решения.
  - Не добавлять декоративные анимации без функциональной пользы.
  - Все новые функции должны работать в интранет/offline-first модели.
  - Все изменения данных делать через аддитивные миграции.
  - Все важные действия писать в аудит.
  - Для массовых операций всегда делать подтверждение.
  - Для импорта всегда делать dry-run перед записью.
  - Для новых UI-разделов сразу добавлять пустые состояния.
  - Для сложных экранов сразу добавлять подсказки и keyboard-friendly управление.

  ———

  ## 9. Ожидаемый результат

  После реализации плана система должна стать не просто доской задач, а полноценным рабочим центром:

  - пользователь утром видит, что делать;
  - руководитель видит риски и нагрузку;
  - администратор управляет системой без ручного вмешательства;
  - задачи меньше теряются;
  - просрочки становятся объяснимыми;
  - повторяющиеся действия автоматизируются;
  - интерфейс остается быстрым и спокойным;
  - система остается локальной, надежной и удобной для интранета.
